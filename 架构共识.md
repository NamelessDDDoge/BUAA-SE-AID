# 三路检测架构共识

## 目的

本文件只记录当前已经确认的内容，不记录仍在讨论中的开放问题。

当前目标是为以下三类任务建立统一架构基础：

1. 图像检测
2. 全文论文检测
3. 同行评审检测

## 当前共识

### 1. 系统采用 5 层理解

当前确认的 5 层如下：

1. 接口层
2. 资源层
3. 编排层
4. 能力层
5. 存储层

它们的职责分别是：

#### 接口层

直接面向前端和管理端接口。

职责：

- 接收上传、建任务、查状态、查结果、下报告
- 返回适合前端展示的数据结构
- 不承担底层检测能力

#### 资源层

负责把用户上传的内容转成系统内部可理解资源。

职责：

- 区分资源类型
- 建立资源之间的关系
- 从文件中提取文本、图像等派生内容
- 为后续编排提供标准化输入

#### 编排层

负责把任务流程串起来。

职责：

- 决定每类任务按什么顺序调用哪些能力
- 管理步骤之间的输入输出衔接
- 负责失败处理、状态推进和流程控制

#### 能力层

负责提供可复用的底层能力。

职责：

- 图像检测能力
- 文本检测能力
- 文本比对/相关性分析能力
- 引文核验能力
- 大模型解释分析能力

这些能力应能被不同任务复用，而不是绑定某一种任务类型。

#### 存储层

负责后端持久化。

职责：

- 存储任务状态
- 存储结构化检测结果
- 存储中间产物索引
- 存储日志、报告和后续审核记录

### 2. 三类任务共享同一架构，不是三套独立系统

图像、论文、同行评审不是三套平行系统，而是三类任务。

三类任务都应经过同一套架构：

```text
接口层 -> 资源层 -> 编排层 -> 能力层 -> 存储层
```

### 3. 图像检测能力必须被论文检测复用

论文检测中的图像分析，不应重新实现一套图像检测逻辑，而应直接复用现有图像检测能力。

### 4. 文本检测能力应被论文检测和同行评审检测共用

论文检测与同行评审检测都需要逐段文本检测，因此应共享同一套文本检测能力。

### 5. 大模型调用能力应统一管理

以下能力都依赖大模型调用：

- 可疑段落解释
- 参考文献相关性分析
- 同行评审与原文逐段相关性分析

因此大模型调用不能散落在业务代码中，后续需要统一纳入能力层，并由管理端统一管理模型配置。

## 论文检测闭环

当前已确认的论文检测流程如下。

### 1. 文件预处理

输入格式包括但不限于：

- PDF
- TXT
- DOCX

处理内容：

- 提取正文文本
- 提取图像

### 2. 文本检测

调用 FastDetectGPT，对论文文本做逐段检测。

模式设计预留两种：

- API 模式
- 本地模式

当前优先级：

- 先实现 API 模式

### 3. 图像检测

对论文中提取出的图像，复用现有图像检测板块。

### 4. 可疑段落分析

对文本检测中可疑的段落，调用大模型 API 做解释分析。

目标：

- 给出某个段落为什么疑似 AI 生成的解释

### 5. 参考文献检测

论文检测闭环中必须包含：

1. 参考文献是否存在
2. 参考文献是否与论文内容对应

## 同行评审检测闭环

当前已确认的同行评审检测流程如下。

### 输入要求

同行评审检测必须同时上传：

1. 原论文文本
2. 评审文本

### 1. 预处理

与论文检测同理，对输入文件做基础解析。

处理内容：

- 提取原论文文本
- 提取评审文本
- 为逐段分析建立结构化文本单元

### 2. 文本检测

与论文检测相同，调用 FastDetectGPT。

模式同样预留：

- API 模式
- 本地模式

当前优先级：

- 先实现 API 模式

### 3. 逐段相关性分析

调用大模型 API，逐段比对同行评审文本与原论文内容是否相关。

目标：

- 判断评审内容是否与原论文内容对应

## 管理端已确认共识

### 1. 日志必须统一成事件模型

日志系统后续必须统一成事件模型，不能继续只做当前这种偏图像和偏用户操作的窄日志。

原因：

- 需要统一覆盖图像、论文、同行评审三类任务
- 需要支撑统计、筛选、追踪和导出
- 需要支撑管理端日志面板与可视化分析

### 2. 管理端需要扩展日志与统计能力

管理端后续需要支持：

- 日志面板扩展
- 数据可视化
- 针对图像、论文、同行评审三类任务的统一日志和统计

### 3. 大模型 API 管理必须预留多协议扩展性

管理端中的模型管理不能只按 OpenAI-compatible 设计。

必须为后续至少预留：

- OpenAI 风格接口
- Anthropic 风格接口
- 完全自定义 HTTP API 接口

即使当前先只实现 OpenAI 风格，也必须在结构上为后续扩展留足空间。

### 4. 模型配置与默认模型选择必须解耦

后续管理端应区分两件事：

1. 维护所有模型配置
2. 为每个功能选择默认模型

也就是说：

- 用户可以先录入一批模型
- API Key 需要隐藏存储
- 在设置默认模型时，只从已有模型中选择

默认模型选择的粒度不是全局唯一一个，而是每个功能各自可选。
